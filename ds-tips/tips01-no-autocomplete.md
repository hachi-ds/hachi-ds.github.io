# AI時代でも必要な「考えて書く」力  
*— VSCode の補完機能をあえて切って学ぶ —*

私は学習のときだけ VSCode の補完機能をオフにしています。  
理由はシンプルで、**主体的にコードを書く力を鍛えるには、あえて不便な環境が必要だから**です。

---

## 学習を阻む“便利すぎる補完”

「自分で書いて考える」ことは、コードへの主体性を育てる上で欠かせません。  
しかし、いざ学習を始めると思わぬ壁にぶつかります。

コードエディタの補完・サジェストがあまりにも優秀で、  
**考える前に答えが提示されてしまう** のです。

私が普段使用している VSCode でも、たとえばこんなことが起きます。

- タイプする前に関数名が先回りして表示される  
- ライブラリ名や引数が自動的に補われてしまう  
- あいまいな理解のままコードが完成してしまう  

便利なはずの機能が、結果として  
**「自分の頭で書く」という学習の核を奪ってしまう** ことがあると感じます。

だからこそ私は、学習時には補完を切るという選択をしています。

---

## 補完をオフにする方法（VSCode）

今回は VSCode を使用する方に向けて、補完機能をオフにする方法を紹介します。  
VSCode はプロジェクトごとに設定を切り替えられるため、学習用ディレクトリだけ不便な環境にすることが可能です。

以下は、私が実際に使用している `settings.json` です。  
プロジェクトのルートディレクトリ直下に `.vscode/settings.json` を作成すれば、補完やサジェストを無効化できます。  
ぜひ学習者の方は活用してみてください。

```
// .vscode/settings.json
{
  // 入力中の候補を一切出さない（コード・コメント・文字列）
  "editor.quickSuggestions": { "other": false, "comments": false, "strings": false },

  // '.' や '(' を打っても候補を出さない
  "editor.suggestOnTriggerCharacters": false,

  // インラインで灰色のゴースト補完を表示しない（Copilot 系も含む）
  "editor.inlineSuggest.enabled": false,

  // 関数呼び出しの引数ヒントを出さない
  "editor.parameterHints.enabled": false,

  // マウスを当てたときの型情報やドキュメントを表示しない
  "editor.hover.enabled": false,

  // ドキュメント内の既出単語からの補完を無効化
  "editor.wordBasedSuggestions": "off",

  // 引数名などの灰色注釈（Inlay Hints）を非表示にする
  "editor.inlayHints.enabled": "off",

  // Enter や Tab で誤って補完を確定しないようにする
  "editor.acceptSuggestionOnEnter": "off",
  "editor.tabCompletion": "off",

  // 電球アイコン（クイックフィックス）を無効化
  "editor.lightbulb.enabled": "off",

  // --- Python 関連 (Pylance) ---
  // 関数補完時に () を自動で付けない
  "python.analysis.completeFunctionParens": false,

  // 自動インポート候補を出さない
  "python.analysis.autoImportCompletions": false
}
```

## Appendix: なぜ“考えて書く力”が必要なのか

そもそも、なぜあえて不便にしてまで「考えて書く力」を鍛える必要があるのでしょうか。

まず前提として、プログラミングにおける「考えて書く力」とは、  
処理の意図を言語化し、必要な構造を選び取り、実装の妥当性を判断して書く力のことだと考えています。  
これは、単にコードを書くだけでなく、問題解決そのものに関わる基礎的なスキルです。

そしてこの能力は、**現代の開発環境では特に軽視されやすい状況があります。**  
なぜなら、AIや自動補完によりコードが自動的に生成されてしまう場面が増えたためです。

これらの技術は実装の効率化に大いに役立ちますが、生成されたコードには次のような問題が発生し得ます。

- プロンプトで意図した指示と異なる設計をしてしまう
- 必要以上に複雑な構造（過剰な構造）を採用してしまう
- 曖昧なままライブラリや処理を選択してしまう
- コードが含む副作用を理解せずに生成してしまう

つまり、自動生成されたコードは便利である一方、**「その実装を選んだ理由」が抜け落ちていることが多い**のです。

そして最終的に人間が問われるのは、

**「なぜこの実装なのか」  
「なぜこの構造にしたのか」**

という説明責任です。  
これは、現在のAIが完全に肩代わりできる領域ではありません。

だからこそ、実装の意図を理解し、自分で判断するための考えて書く力が必要になります。  
その力を鍛えるために、あえて不便な環境で学習するという選択が意味を持つのです。

学ぶときは不便に、
作るときはAIの力を借りて効率化する。  

そんな“使い分け”が、これからのデータサイエンティストに求められる姿だと感じています。
